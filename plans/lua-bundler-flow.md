# Lua Bundler Flow Plan

## Goal

Produce a single self-contained Lua file that can live alongside the TourGuide authoring skill at `skills/tourguide/tourguide.lua`.

The bundled file should let the skill distribute or load `tourguide.nvim` without requiring the full plugin directory layout.

## Current State

- Runtime modules live under `lua/tourguide/`.
- The plugin entrypoint is `plugin/tourguide.lua` and calls `require("tourguide").setup()`.
- The skill lives at `skills/tourguide/SKILL.md`.
- There is no current build, package, or bundler tooling.
- The plugin has no external runtime dependencies.

## Design

Add a small repository-local bundler script that emits one Lua file by registering each module with `package.preload`.

The generated file should roughly follow this shape:

```lua
if vim.g.loaded_tourguide_nvim == 1 then
  return
end

package.preload["tourguide.state"] = function()
  -- contents of lua/tourguide/state.lua
end

package.preload["tourguide.loader"] = function()
  -- contents of lua/tourguide/loader.lua
end

-- remaining modules

vim.g.loaded_tourguide_nvim = 1
require("tourguide").setup()
```

This preserves the existing `require("tourguide.*")` relationships while allowing the plugin to run from a single file.

## Files To Add

- `scripts/bundle.lua`
- `Makefile`
- `skills/tourguide/tourguide.lua` generated by the bundler

## Bundler Details

The bundler should:

- Use an explicit module order so output is deterministic.
- Read each `lua/tourguide/*.lua` file.
- Wrap each module as `package.preload["module.name"] = function() ... end`.
- Include a generated-file header that says not to edit it manually.
- Include the plugin loader behavior from `plugin/tourguide.lua` or equivalent inline code.
- Write to `skills/tourguide/tourguide.lua`.
- Avoid external dependencies.

Suggested module order:

```lua
{
  "tourguide.state",
  "tourguide.resolve",
  "tourguide.annotations",
  "tourguide.sidebar",
  "tourguide.layout",
  "tourguide.navigation",
  "tourguide.loader",
  "tourguide.init",
}
```

The exact order does not need to match runtime dependency order because `package.preload` functions execute lazily, but keeping it explicit makes diffs stable.

## Make Target

Add a `Makefile` target:

```make
bundle:
	@nvim --headless -u NONE -l scripts/bundle.lua
```

If plain `lua` is sufficient after implementation, prefer that. If the bundler needs `vim.fn` utilities, use headless Neovim.

## Documentation Updates

Update:

- `README.md` with `make bundle` usage.
- `doc/tourguide.txt` with bundled-file notes.
- `skills/tourguide/SKILL.md` to mention `tourguide.lua` exists beside the skill.

## Verification

Run:

```sh
make bundle
nvim --headless -u NONE -l skills/tourguide/tourguide.lua -c 'qa!'
nvim --headless -u NONE -c 'luafile skills/tourguide/tourguide.lua' -c 'TourGuide tours/solid_effects_async.lua' -c 'TourGuideNext' -c 'TourGuideClose' -c 'qa!'
```

The second command may need adjustment because `-l` exits differently from `:luafile`; the important smoke test is loading the bundled file and proving the commands are registered.

## Risks

- Generated output may be noisy in diffs if file order is not stable.
- The bundled plugin and source modules can drift if contributors forget to run `make bundle`.
- Headless tests may need special handling once first-run keybind prompts are added.

## Non-Goals

- Do not introduce a third-party Lua bundler.
- Do not minify the generated Lua.
- Do not change how normal plugin installs work.
